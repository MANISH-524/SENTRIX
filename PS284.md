# SENTRIX — PS284: Recovery Readiness Agent

> **Create a recovery readiness agent that checks backup status, tests restore
> evidence, and flags recovery-point risks.**
> Inputs: backup logs, restore test records, asset criticality.
> Deliverable: recovery readiness dashboard.

---

## The one-sentence pitch

> Every backup dashboard tells you your backups **succeeded**. None of them can
> tell you whether you can actually **recover** — because nobody models the
> decay of restore evidence. SENTRIX scores **provable recoverability**, and
> when it doesn't know, it **schedules the test that finds out.**

---

## What makes this different

Veeam ONE, Rubrik Radar, Commvault, Cohesity and Druva all ship backup health
scoring and SLA dashboards. We are not claiming "nobody built a backup
dashboard." We are claiming something narrower and defensible:

> They monitor whether **backups succeeded**.
> They do not model whether **restores are provable**.

A successful backup job is a **claim**. A passed restore test is **evidence**.
Every commercial tool treats the claim as the truth. PS284's middle clause —
*"tests restore evidence"* — is the gap, and taking those three words seriously
produces an architecture that does not currently exist.

### The reframe

**Recovery readiness is not a monitoring problem. It is an evidence-provability
problem under decay.**

Three consequences:

1. **Evidence decays.** A restore test that passed 214 days ago proves far less
   today — configs drifted, dependencies moved, credentials and KMS keys
   rotated, schemas migrated. Restorability confidence is a *decaying function
   of evidence age*, not a boolean.

2. **Confidence is falsifiable, not asserted.** Every asset carries a **Proven
   Recovery Confidence (PRC)** score with an explicit provenance chain: what
   evidence, how old, how decayed, what's missing.

3. **The agent reduces its own uncertainty.** A monitoring tool reports "no
   restore test in 214 days" and stops. SENTRIX recognises its confidence is low
   *because evidence is missing*, then **schedules the restore test that
   manufactures the missing evidence** — and re-scores when the result lands.

That third point is the thesis. It is what makes this an **agent** rather than a
dashboard with an LLM bolted on:

> **An agent that knows what it doesn't know about recoverability,
> and takes action to find out.**

### The demo that proves it

```
payments-db  — backup 2h old against a 4h RPO, zero consecutive failures
               → GREEN on every commercial dashboard
               → SENTRIX: Proven Recovery Confidence 8.8% (blind spot)
                  · last proven restore 214d ago
                  · tier-1 evidence half-life is 30d → decayed to 0.7% strength
                  · 47 config changes since that restore
                  · "You have not proven you can recover this in over 7 months."
               → Restore test scheduled tonight (highest confidence gain in fleet)
```

That gap between "green" and "8.8%" is the entire project.

## Architecture

```
  Evidence ledger  ──►  Confidence engine  ──►  Dual reasoning
  (backups,             (decay + drift,          (LLM proposes,
   restore tests,        deterministic)           floor disposes)
   criticality)                                        │
        ▲                                              ▼
        │                                     Evidence acquisition
        └──────── test result written back ◄──  (schedules the test
                                                  that cuts uncertainty)
                                                        │
                                                        ▼
                                              Readiness dashboard
```

The **feedback arrow is the novelty.** Commercial tools stop at the dashboard.

### 1. Evidence ledger — `agent/recovery/evidence_ledger.py`

PS284 hands you "restore test records" as an input. Reducing that to a
`days_overdue` integer throws away everything that makes it evidence. A record is:

```json
{"asset_id": "payments-db", "test_type": "full_restore_drill",
 "outcome": "passed", "started_at": "...", "rto_actual_seconds": 900,
 "bytes_restored": 402653184, "checksum_verified": true, "signature": "..."}
```

Because those fields answer what a counter cannot: **what** was proven, did it
meet the **RTO target**, was data **verified**, and — most valuable — did it
**fail**? A failed drill is proof of non-recoverability; conventional tools
discard it.

Append-only JSONL, HMAC-signed with the audit key. Restore evidence is exactly
what gets quietly edited after a bad audit, so it is tamper-evident by
construction. Records failing signature verification are excluded and reported,
never silently trusted.

### 2. Confidence engine — `agent/recovery/confidence.py`

```
PRC(asset) = 0.30 · BackupFreshness(hours_since / rpo_target)
           + 0.50 · EvidenceStrength(test_type) · exp(−λ_tier · days_since_test)
           + 0.20 · ChainIntegrity(consecutive_failures)
           −        DriftPenalty(changes_since_last_proven_restore)
```

Three novel claims:

| Term | Claim | Why it matters |
|---|---|---|
| `exp(−λ·days)` | Evidence decays exponentially, and **λ is per-tier** | A tier-1 transactional DB drifts fast (30d half-life); a cold archive slowly (180d). Identical evidence ages differently. |
| `EvidenceStrength` | A checksum verify (0.30) ≠ a partial restore (0.65) ≠ a full drill (1.00) | Nobody grades evidence *quality*. |
| `DriftPenalty` | Count config changes since the last **proven** restore | A backup whose source changed 47 times since the last proven restore is untrustworthy. **This explains why all-green backups fail during real incidents.** Saturating (Michaelis–Menten), ceiling 0.35. |

Fleet confidence is **criticality-weighted**: a proven cold archive must not
offset an unproven payments DB.

**Determinism contract:** pure arithmetic over ledger facts. No LLM touches the
number. The LLM *explains* the score; it never *computes* it. The LLM's failure
mode is "unhelpful explanation," never "wrong confidence."

**Confidence intervals:** reported as `31% ± 12%`, widening as evidence goes
stale. An agent that says "I don't know how recoverable this is, and here's how
unsure I am" is more trustworthy than one asserting a point estimate.

### 3. Dual reasoning — `agent/reasoning/reasoning_core.py`

Already in the codebase, and the answer to *"what if your LLM hallucinates?"*:

> The LLM may **escalate**. The LLM may **never de-escalate** below the
> deterministic floor.

`_hard_floor_action()` guarantees 3+ consecutive failures on a tier-1 asset is
always at least a P1 — regardless of what the model says. Combined with
`ai_safety.validate_assessments()`'s closed action allow-list, this is a
**bounded-authority agent**: it can only ever raise an alarm, never silence one.

### 4. Evidence acquisition — `agent/recovery/evidence_scheduler.py`

**The agentic component.** You cannot restore-test everything every week. Given
a test budget, *which* tests most reduce fleet-wide recovery uncertainty?

Two-tranche scheduling:

- **Tranche 1 (mandatory):** every tier-1/2 asset in a `blind_spot`/`unproven`
  band, ordered by criticality.
- **Tranche 2 (opportunistic):** remaining budget goes to greedy
  `Δ(criticality-weighted confidence) / test_cost`.

Why two tranches — a real bug we found and fixed:

> Pure value-per-hour greedy scheduled **15 tier-4 checksum verifies (IoT hubs,
> laptops) and deferred an unproven tier-1 database.** Cheap tests win on ratio.
> But you cannot offset an unproven payments DB with fifteen proven laptops —
> aggregate confidence hides categorical risk.

Pinned by `test_scheduler_covers_critical_blind_spots_before_cheap_wins`.

Greedy is deliberate: O(n log n), explainable line-by-line to an auditor, and
within (1−1/e) of optimal for a submodular gain function. An unexplainable
optimum is worth less here than a defensible near-optimum.

Output is not an alert — it's a **work queue that provably shrinks the blind
spot**, and it reports **unfunded critical work** explicitly:

```
Scheduled 2 restore test(s) (2 on critical tier-1/2 assets) using 12.0h of a 12h
budget. If all pass, fleet recovery confidence rises 55.9% → 57.0% and blind
spots fall 71 → 69. WARNING: 21 critical tier-1/2 asset(s) remain unproven and
unfunded — 126.0h more budget would cover them.
```

An agent that silently drops a tier-1 blind spot when the budget runs out is
lying by omission.

### 5. Readiness dashboard — `dashboard/src/components/RecoveryReadiness.js`

Not "94% of backups succeeded." Per asset:

- **Proven Recovery Confidence** with interval, and the band (proven / probable
  / unproven / blind spot)
- **Provenance**: what test, how many days ago, decayed to what % of original
  strength, how much drift since, whether the RTO target was met
- **Contribution breakdown**: every weighted term, including the negative drift
  penalty
- **Decay curve**: confidence falling from the last test toward the unproven
  threshold, with "today" marked — *the chart that sells the project*
- **The agent's plan**: which tests, which tranche, cost, projected gain
- **Record a test** → confidence re-scores instantly. The loop closes on screen.

## Accuracy and reliability

Uniqueness is worthless if it isn't defensible. Four properties:

**Determinism where it matters.** PRC is a closed-form function of ledger facts —
reproducible, unit-tested (`test_scoring_is_deterministic`), identical across
runs. The LLM never computes it.

**Bounded LLM authority.** Hard floor + closed action allow-list +
dry_run/approve/auto modes. Unknown actions from the model downgrade to
`MANUAL_REVIEW` — surfaced to a human, never routed to the executor.

**Graceful degradation.** Zero LLM providers reachable → the rule engine still
scores, still schedules, still alerts. Demonstrated live: kill the API key and
the cycle completes.

**Falsifiability.** λ is the model's central claim, and it is **a guess until
calibrated.** We say so. `calibrate_lambda()` fits it by least squares on
observed restore outcomes bucketed by evidence age, and reports sample size so a
thin sample is labelled thin rather than silently trusted.
`test_calibration_recovers_a_known_lambda` synthesises decay with a known 40-day
half-life and verifies the fitter recovers it within 25%.

> Admitting an unvalidated parameter and showing the experiment that validates it
> reads as rigour. Claiming a magic number reads as fake.

---

## Honest limitations

- **λ defaults are informed priors, not measurements.** Calibrate against your
  own restore history before trusting the absolute numbers. The *ranking* is
  robust well before the absolute score is.
- **Drift is currently a proxy** (tier-based change rate × evidence age). Wiring
  a real change feed — CMDB, git, Terraform state, deploy events — makes this
  the strongest signal in the model rather than the most assumed.
- **Simulated evidence is clearly labelled.** `evidence_source: "simulated"` vs
  `"ledger"`. A fresh install demonstrates the model; real records take over the
  moment they exist. Simulated evidence is never presented as proof.
- **Test costs are configuration, not physics.** `TEST_COST_HOURS` should
  reflect your environment.
